# 新手必读：这套环境到底是干什么的？

> 如果你不太了解 Git、Jenkins、Docker、K8s 分别有什么用，**先读本文，再动手部署**。  
> 部署操作步骤请看 [DEPLOYMENT_GUIDE.md](./DEPLOYMENT_GUIDE.md)。

---

## 一、用一句话说清楚

你要搭建的是一条 **「代码自动变成线上服务」的流水线**：

```
你改代码 → 提交到 Git 仓库 → 机器自动编译、打包 → 自动部署到服务器上运行
```

以前可能要：手动编译 → 手动拷 jar 包 → 手动重启服务。  
搭好之后：**git push 一下，等几分钟，新版本就上线了**。

---

## 二、打个比方（整条流水线）

可以把整个过程想象成 **开餐馆**:

| 现实世界 | 对应技术 | 干什么 |
|----------|----------|--------|
| **菜谱和配方本** | Git + GitHub/Gitea | 存放所有源代码，记录每次改了什么 |
| **中央厨房 + 流水线** | Jenkins | 收到通知后自动按步骤做菜（编译、打包） |
| **标准餐盒** | Docker 镜像 | 把做好的菜装进统一规格的盒子，到哪都能热 |
| **餐盒仓库** | 阿里云镜像仓库 | 临时存放做好的餐盒，方便取用 |
| **餐厅大厅和桌位** | K3s (Kubernetes) | 真正接待顾客的地方，运行你的应用 |
| **服务员** | Pod（容器实例） | 实际在跑程序的一个个实例 |

**Webhook** 就像「门铃」：你一 push 代码，Git 仓库按一下门铃，Jenkins 听到就开始干活。

## 三、每个组件详细说明

### 3.1 Git —— 代码的版本管理工具

**是什么：** 安装在电脑上的命令行工具，帮你管理代码变更。

**有什么用：**
- 记录每次修改（commit）
- 可以回退到历史版本
- 多人协作不互相覆盖

**你需要做什么：** 在 Windows 开发机上安装 Git，用 `git add`、`git commit`、`git push` 提交代码。

**不装会怎样：** 无法把代码上传到仓库，Jenkins 也拉不到代码。

---

### 3.2 GitHub / Gitea —— 远程代码仓库

**是什么：** 存放代码的「网盘 + 版本历史」，带网页界面。

| 类型 | 说明 |
|------|------|
| **GitHub (github.com)** | 公有云服务，代码在 GitHub 公司服务器上 |
| **Gitea（自建）** | 装在你自己机器上的 Git 服务，像「公司内部的 GitHub」 |
| **GitHub Enterprise** | GitHub 官方企业版，需单独服务器，配置要求高 |

**有什么用：**
- 集中存放项目源码
- 提供 Web 界面看代码、提 Issue
- **通知 Jenkins**：「代码更新了，快来构建！」（Webhook）

**你需要做什么：** 创建仓库，把项目 push 上去。

**不装会怎样：** 代码没地方放，CI/CD 没有起点；Jenkins 不知道要构建哪个项目。

---

### 3.3 Jenkins —— 自动化构建与部署（CI/CD 核心）

**是什么：** 一个跑在服务器上的 **自动化任务调度器**，有网页控制台。

**CI** = Continuous Integration（持续集成）= 代码一提交就自动编译、测试  
**CD** = Continuous Deployment/Delivery（持续部署/交付）= 编译通过后自动部署上线

**有什么用：**
- 自动从 Git 拉最新代码
- 自动执行 `mvn package` 等编译命令
- 自动构建 Docker 镜像并推送
- 自动调用 K8s 更新线上版本
- 构建失败发通知、保留日志

**你需要做什么：** 安装 Jenkins，配置流水线（Jenkinsfile），连接 Git 和 K8s。

**不装会怎样：** 每次改代码都要手动编译、手动部署，容易出错、效率低。

**访问方式：** 浏览器打开 `http://服务器IP:8080`

---

### 3.4 Docker —— 容器化打包工具

**是什么：** 把应用和运行环境（Java、依赖库、配置）打成一个 **镜像（image）**。

**有什么用：**
- **环境一致**：开发机、测试机、生产机跑的是同一个镜像，避免「我这边能跑你那边不行」
- **部署简单**：不用在服务器上装 JDK、配环境变量，拉镜像就能跑
- **隔离**：多个应用互不干扰

**类比：** 镜像 = 光盘里的完整游戏安装包；容器 = 正在运行的一个游戏实例。

**你需要做什么：** 编写 `Dockerfile`（打包说明书），Jenkins 构建时自动执行。

**不装会怎样：** 无法标准化打包，K8s 也难以管理应用。

---

### 3.5 阿里云镜像仓库 (ACR) —— 镜像的「中转站」

**是什么：** 专门存放 Docker 镜像的云服务（类似 Docker 版的 GitHub）。

**有什么用：**
- Jenkins 构建完镜像 **push** 到这里
- K3s 部署时从这里 **pull** 镜像
- 国内访问比 Docker Hub 快且稳定

**为什么需要中转：** Jenkins 和 K3s 可能在同一台机器，但仍建议用仓库：
- 保留历史版本（方便回滚）
- 多台 K8s 节点都能拉同一个镜像
- 镜像有权限控制

**不配置会怎样：** 镜像只能存在本机，不便管理和回滚；国内 pull 公网镜像可能很慢。

---

### 3.6 K3s (Kubernetes) —— 应用运行平台

**是什么：** 轻量版的 **Kubernetes（K8s）**，负责在服务器上 **运行、管理、扩缩容** 你的应用容器。

**有什么用：**
- 自动拉起应用（挂了会重启）
- 滚动更新（新版本逐步替换旧版本，减少停机）
- 统一管理多个服务（前端、后端、数据库代理等）
- 配置 Service、Ingress 对外提供访问

**K3s 和完整 K8s 的区别：** K3s 更省内存，适合 4GB 小服务器；功能对中小型项目够用。

**常用概念：**

| 概念 | 含义 |
|------|------|
| **Pod** | 最小运行单位，通常一个 Pod 跑一个应用容器 |
| **Deployment** | 描述「要跑几个副本、用什么镜像」 |
| **Service** | 给 Pod 一个稳定的内部访问地址 |
| **Namespace** | 命名空间，用来隔离不同项目（如 `enterprise-platform`） |
| **kubectl** | 操作 K8s 的命令行工具 |

**不装会怎样：** 只能手动 `docker run` 跑容器，无法自动更新、自动恢复，运维成本高。

## 四、完整流程：从改代码到上线

假设你改了一行业务代码：

```
第 1 步  【你的电脑】
         git add . && git commit -m "修复登录bug" && git push
         → 代码上传到 Git 仓库

第 2 步  【Git 仓库】
         检测到 push，发出 Webhook 通知
         → 「嘿 Jenkins，main 分支有新代码！」

第 3 步  【Jenkins】
         ① Checkout：从 Git 拉最新代码
         ② Build：mvn package 编译成 jar
         ③ Docker Build：按 Dockerfile 打成镜像
         ④ Docker Push：镜像推到阿里云 ACR
         ⑤ Deploy：kubectl 告诉 K3s 用新镜像

第 4 步  【K3s】
         拉取新镜像 → 启动新 Pod → 健康检查通过 → 停掉旧 Pod
         → 用户访问到的已是新版本

第 5 步  【你】
         打开 Jenkins 看构建是否成功（蓝色=成功，红色=失败）
         打开应用验证功能是否正常
```

**全程你只需：** 写代码 + push + 看结果。中间编译、打包、部署全自动。

## 五、部署文档里每一步是在干什么？

下面把 [DEPLOYMENT_GUIDE.md](./DEPLOYMENT_GUIDE.md) 各章节和「有什么用」对应起来：

| 章节 | 安装/配置什么 | 为什么要做 | 不做会怎样 |
|------|---------------|------------|------------|
| **第 4 章** Git + GitHub | Git 工具、代码仓库 | 代码的「源头」 | Jenkins 无代码可构建 |
| **第 5 章** 上传 deploy | 部署脚本到服务器 | 自动化安装 | 只能纯手工装，易出错 |
| **第 6 章** 系统初始化 | apt 源、swap、防火墙 | 系统能稳定跑服务 | 内存不够崩溃、端口不通 |
| **第 7 章** Docker | 容器引擎 | 构建和运行镜像 | 无法 container 化部署 |
| **第 8 章** K3s | 容器编排平台 | 自动运行和更新应用 | 只能手动 docker run |
| **第 9 章** Jenkins | CI/CD 引擎 | 串联自动化流水线 | 无法自动构建部署 |
| **第 10 章** Jenkins 控制台 | 插件、管理员 | 能用网页配置任务 | Jenkins 无法正常工作 |
| **第 11 章** GitHub 集成 | Token、Webhook | push 后自动触发构建 | 只能手动点「构建」 |
| **第 12 章** 阿里云 ACR | 镜像仓库 | 存镜像、供 K3s 拉取 | 镜像管理混乱 |
| **第 13 章** Pipeline | Jenkinsfile | 定义构建部署步骤 | Jenkins 不知道执行什么 |
| **第 14 章** 验证 | 端到端测试 | 确认整条链路通了 | 可能某环节 silently 失败 |

## 七、两种 Git 方案怎么选？

| 方案 | 适合谁 | 说明 |
|------|--------|------|
| **GitHub.com（公网）** | 个人学习、开源项目 | 免费，文档默认方案；国内可能较慢 |
| **Gitea 自建（同机）** | 公司内网、4GB 小服务器 | 代码在自己机器上，Webhook 走内网更简单 |
| **GitHub Enterprise** | 大型企业 | 需独立高配服务器，4GB 同机不推荐 |

若你公司 Git 地址是自己的服务器（如 `git.xxx.com`），就是 **自建 Git**，流程一样，只是地址和插件可能从 GitHub 换成 Gitea/GitLab。

## 八、4GB 内存为什么容易紧张？

同一台机器可能要同时跑：

```
Gitea/Git  ≈ 300MB
Jenkins    ≈ 512MB
K3s        ≈ 800MB
Docker构建  ≈ 临时占用 500MB+
系统本身    ≈ 500MB
────────────────────
合计        接近或超过 4GB
```

所以文档里会：
- 配置 **4GB swap**（用硬盘当临时内存）
- 限制 Jenkins 内存 **-Xmx512m**
- 选用 **K3s** 而不是完整 K8s
- 建议给 Pod 设置资源上限

## 十、名词速查表

| 名词 | 通俗解释 |
|------|----------|
| **仓库 (Repository)** | 存放一个项目代码的地方 |
| **commit** | 一次代码提交记录 |
| **push** | 把本地提交上传到远程仓库 |
| **clone** | 把远程仓库复制到本地 |
| **Webhook** | 仓库在有事发生时通知别的系统（如 Jenkins） |

| **镜像 (Image)** | 应用的只读打包文件 |

| **Token** | 访问 Git 服务的密码牌（代替账号密码） |
| **RBAC** | 谁能在 K8s 里做什么操作的权限规则 |
| **滚动更新** | 新版本逐步替换旧版本，减少停机 |
| **CI/CD** | 持续集成 / 持续部署，自动化构建和上线 |

## 十二、常见问题（概念层面）

**Q：我能不能不用 Docker 和 K8s，只用 Jenkins？**  
可以 Jenkins 编译完直接 scp jar 到服务器重启，但环境不一致、难回滚、难扩展。Docker+K8s 是现代化标准做法。

**Q：Jenkins 和 K3s 必须同一台机器吗？**  
不必须。小项目可以同机；大了可以 Jenkins 一台、K8s 集群多台。

**Q：GitHub 和 Git 是一回事吗？**  
不是。Git 是工具；GitHub/Gitea 是用 Git 协议的远程托管服务。

**Q：构建失败了看谁？**  
先看 **Jenkins 控制台输出**（Console Output），里面会写哪一步错了。

**Q：应用访问不了看谁？**  
`kubectl get pods` 看 Pod 是否在 Running，再看 `kubectl logs` 看应用日志。
